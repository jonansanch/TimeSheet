# Story 6.2: Consultar Bitácora como Admin

Status: review

## Story

As an admin,
I want consultar y filtrar la bitácora completa del sistema,
So that pueda responder auditorías y revisar acciones sensibles.

## Acceptance Criteria

1. **Given** un Admin autenticado, **when** abre `/admin/bitacora`, **then** ve un panel de filtros (fecha desde/hasta, usuario, tipo de evento) **and** una tabla `MudDataGrid` con los eventos registrados en `BitacoraAuditoria`.

2. **Given** resultados de bitácora, **when** se muestran en tabla, **then** cada fila muestra: timestamp formateado (`dd/MM/yyyy HH:mm`), tipo de evento (como chip de color), actor (nombre o email), entidad afectada, ID de entidad y metadata (truncada con tooltip para el JSON completo).

3. **Given** filtros aplicados, **when** el Admin hace clic en "Buscar", **then** el frontend llama `GET /api/bitacora` con los parámetros de query correspondientes **and** la tabla se actualiza con los resultados.

4. **Given** filtros que no producen resultados, **when** la API retorna lista vacía, **then** se muestra un estado vacío con mensaje descriptivo **and** los filtros se conservan para ajustes.

5. **Given** un usuario no-Admin, **when** intenta acceder a `GET /api/bitacora` o a `/admin/bitacora`, **then** recibe `403 Forbidden` (backend) o es redirigido (frontend `[Authorize(Roles = Roles.Admin)]`).

6. **Given** la tabla BitacoraAuditoria, **when** se consulta vía API, **then** el endpoint solo permite GET (sin POST, PUT, DELETE) **and** retorna máximo 500 eventos ordenados por Timestamp DESC.

- [x] Crear `Backend/src/Infrastructure/Bitacora/GetBitacoraQueryHandler.cs`
  - Inyecta `IDbConnection db` (Dapper — igual que `GetHistorialNotificacionesQueryHandler`)
  - SQL:
    ```sql
    SELECT b.Id,
           b.TipoEvento,
           b.ActorId,
           b.ActorEmail,
           ISNULL(u.NombreCompleto, b.ActorEmail) AS ActorNombre,
           b.EntidadAfectada,
           b.EntidadId,
           b.Timestamp,
           b.MetadataJson
    FROM   BitacoraAuditoria b
    LEFT   JOIN AspNetUsers u ON u.Id = b.ActorId
    WHERE  (@Desde      IS NULL OR CAST(b.Timestamp AS date) >= @Desde)
      AND  (@Hasta      IS NULL OR CAST(b.Timestamp AS date) <= @Hasta)
      AND  (@ActorId    IS NULL OR b.ActorId = @ActorId)
      AND  (@TipoEvento IS NULL OR b.TipoEvento = @TipoEvento)
    ORDER  BY b.Timestamp DESC
    OFFSET 0 ROWS FETCH NEXT 500 ROWS ONLY
    ```
  - Retorna `new BitacoraResponse(rows.Count, rows)`

### Backend — Endpoint API

- [x] Crear `Backend/src/Api/Endpoints/Bitacora.cs`
  ```csharp
  public class Bitacora : IEndpointGroup
  {
      public static string RoutePrefix => "/api/bitacora";

      public static void Map(RouteGroupBuilder groupBuilder)
      {
          var adminOnly = new AuthorizeAttribute { Roles = Roles.Admin };
          groupBuilder.MapGet("", GetBitacora).RequireAuthorization(adminOnly);
      }

      public static async Task<IResult> GetBitacora(
          ISender sender, CancellationToken cancellationToken,
          [FromQuery] DateOnly? desde = null,
          [FromQuery] DateOnly? hasta = null,
          [FromQuery] string? actorId = null,
          [FromQuery] string? tipoEvento = null)
      {
          var query = new GetBitacoraQuery(desde, hasta, actorId, tipoEvento);
          var result = await sender.Send(query, cancellationToken);
          return Results.Ok(result);
      }
  }
  ```

### Parámetro ActorId vs UserId

El parámetro de query en la API y en el repositorio frontend se llama `actorId` (nombre de la columna en `BitacoraAuditoria`) para diferenciarlo del `userId` que usan otros filtros. El frontend lo pasa directamente desde el Id del usuario seleccionado en el dropdown.

### Mapeo Dapper: ActorNombre

`ActorNombre` es una columna calculada en el SQL (`ISNULL(u.NombreCompleto, b.ActorEmail) AS ActorNombre`). Dapper la mapea automáticamente al campo `ActorNombre` del record `BitacoraItemDto` porque coincide el nombre. Si el usuario no existe en `AspNetUsers` (e.g., fue eliminado), retorna `b.ActorEmail`.

### Timestamp: DateTimeOffset → LocalDateTime

El campo `Timestamp` se almacena como UTC (`DateTimeOffset` con offset `+00:00`). Para mostrar al usuario la hora local del servidor se usa `.LocalDateTime`. Si el servidor corre en la zona horaria de Colombia (UTC-5), `.LocalDateTime.ToString("dd/MM/yyyy HH:mm")` mostrará la hora local. Esto es consistente con cómo lo hace `NotificacionesPage.razor`:

```razor
@context.Item.FechaEnvio.LocalDateTime.ToString("dd/MM/yyyy HH:mm")
```
