# Story 4.3: Gestionar Empleados

Status: done

## Story

As an admin,
I want crear, consultar, actualizar y desactivar empleados,
so that el sistema tenga un catalogo confiable de recursos que alimente los registros de horas y los reportes.

## Acceptance Criteria

1. **Given** un Admin autenticado, **when** abre `/admin/empleados`, **then** ve una tabla con todos los empleados (activos e inactivos), su nombre, estado y acciones disponibles.

2. **Given** la tabla de empleados, **when** se renderiza, **then** soporta paginacion y busqueda por nombre sin recargar la pagina completa.

3. **Given** datos validos de nuevo empleado, **when** el Admin lo crea, **then** el sistema persiste el empleado con `Activo = true` y lo muestra en la tabla.

4. **Given** datos invalidos de nuevo empleado (nombre vacio o mayor a 200 caracteres), **when** el Admin intenta guardar, **then** el API responde con Problem Details/Validation y la UI muestra errores especificos por campo sin cerrar el formulario.

5. **Given** un empleado activo, **when** el Admin lo desactiva, **then** el empleado pasa a `Activo = false`, desaparece del selector de recurso en el formulario de nuevo registro y los registros historicos con ese recurso se conservan intactos.

6. **Given** un empleado inactivo, **when** el Admin lo activa, **then** `Activo = true` y vuelve a aparecer en el selector de recurso del formulario de registro.

7. **Given** un empleado existente, **when** el Admin edita su nombre con datos validos, **then** el nombre se actualiza en la tabla sin afectar los registros historicos que ya usaron ese valor.

8. **Given** el formulario de registro de horas (`KpgShiftForm`), **when** el usuario elige el campo Recurso, **then** el selector muestra unicamente los empleados activos obtenidos desde la API en lugar de la lista hardcodeada.

## Dev Notes

### Empleado vs ApplicationUser

`Empleado` es una entidad de catalogo de dominio independiente de `ApplicationUser` (Identity). Representa el recurso que aparece en los registros de horas (campo `Recurso`). Un usuario del sistema puede o no estar asociado a un empleado del catalogo; esa relacion no se formaliza en esta story.

`RegistroHoras.Recurso` se mantiene como `string` para no violar la inmutabilidad de registros existentes. El catalogo de empleados alimenta el dropdown del formulario, pero los registros historicos conservan el string que se guardo originalmente.

### Seed inicial

Los tres recursos actualmente hardcodeados en `KpgShiftForm` deben existir en la BD como empleados activos:
- "Consultor"
- "Analista"
- "Lider tecnico"

El seed se ejecuta en `ApplicationDbContextInitialiser.SeedAsync()` solo si la tabla `Empleados` esta vacia, igual que el seed de usuarios.

### Indice unico en Nombre

Para evitar duplicados en el catalogo, agregar indice unico en `Empleados.Nombre`. Si el Admin intenta crear un empleado con nombre ya existente (activo o inactivo), el error de BD debe capturarse y devolver un mensaje amigable via Problem Details (sin exponer el mensaje raw de SQL Server).

En el handler de `CreateEmpleado`, capturar `DbUpdateException` cuya `InnerException` contenga "UNIQUE" o "duplicate" y convertirla a `ValidationException` con mensaje: `"Ya existe un empleado con ese nombre."`.

### Desactivacion y registros historicos

Cuando un empleado se desactiva (`Activo = false`):
- No se modifica ningun `RegistroHoras` existente.
- El campo `RegistroHoras.Recurso` sigue siendo el string original.
- El empleado desaparecera del dropdown en nuevos registros porque `GetActivosAsync` filtra `soloActivos: true`.
- La tabla admin sigue mostrando todos (activos e inactivos) para poder reactivar.

### Navegacion Admin

La seccion Admin en el menu lateral actualmente solo tiene "Usuarios" y "Solicitudes". Agregar "Empleados" como nuevo item. Verificar si existe un componente de navegacion reutilizable o si cada item se agrega manualmente.

### Patron de repository frontend

Seguir exactamente el mismo patron de `UserAdminRepository`:
- Autenticacion via `AuthStateService.GetTokenAsync()`.
- Headers `Authorization: Bearer {token}`.
- Manejo de errores con `ReadErrorAsync` para extraer `ProblemDetails` y `ValidationProblemDetails`.
- No lanzar excepciones al componente; retornar tuplas `(bool Ok, T? Data, string? Error)`.
