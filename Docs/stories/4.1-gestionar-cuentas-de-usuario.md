# Story 4.1: Gestionar Cuentas de Usuario

Status: done

## Story

As an admin,
I want crear, activar, desactivar y eliminar cuentas de usuario,
so that pueda controlar quien accede a KPG Timesheet.

## Acceptance Criteria

1. **Given** un Admin autenticado, **when** abre `/admin/usuarios`, **then** ve una tabla con usuarios, email, estado, rol principal y acciones disponibles.

2. **Given** la tabla de usuarios, **when** se renderiza, **then** permite paginacion y ordenamiento por email/estado/rol sin mostrar datos sensibles como password hash, security stamp, refresh tokens o claims internos.

3. **Given** datos validos de un nuevo usuario, **when** el Admin crea la cuenta, **then** Identity crea el usuario con password hasheado, email como username, estado activo y rol inicial valido.

4. **Given** datos invalidos de nuevo usuario, **when** el Admin intenta crear la cuenta, **then** el API responde con Problem Details/Validation y la UI muestra errores claros sin cerrar el formulario.

5. **Given** una cuenta activa existente, **when** el Admin la desactiva, **then** el usuario pierde acceso en el siguiente login/refresh y no se eliminan registros historicos asociados.

6. **Given** una cuenta inactiva existente, **when** el Admin la activa, **then** el usuario puede iniciar sesion nuevamente si sus credenciales son validas.

7. **Given** una cuenta existente sin datos historicos asociados, **when** el Admin confirma eliminar, **then** el sistema puede eliminar fisicamente la cuenta de Identity.

8. **Given** una cuenta existente con registros historicos o solicitudes asociadas, **when** el Admin confirma eliminar, **then** el sistema conserva la cuenta como inactiva para no romper trazabilidad historica.

9. **Given** el Admin intenta desactivar o eliminar su propia cuenta autenticada, **when** la solicitud llega al API, **then** el sistema la rechaza con 400/403 para evitar dejar al sistema sin administrador operativo.

## Dev Notes

### Alcance exacto de Story 4.1

Esta story abre la Epic 4 con cuentas de usuario. Incluye alta, activacion, desactivacion, eliminacion y listado. **No** incluye cambio de rol para usuarios existentes; eso corresponde a Story 4.2. La unica asignacion de rol en esta story es el rol inicial al crear una cuenta.

### Estado actual del codigo

`Backend/src/Api/Endpoints/Users.cs` existe pero no mapea endpoints. Es el punto natural para esta story.

`ApplicationUser` actualmente hereda de `IdentityUser` sin propiedades adicionales. Para activar/desactivar sin borrar trazabilidad, agregar `IsActive`, `DeactivatedAt`, `DeactivatedBy`.

`IdentityService` ya tiene:
- `CreateUserAsync(string userName, string password)`;
- `DeleteUserAsync(string userId)`;
- `ValidateCredentialsAsync(email, password)`;
- `ValidateCredentialsByIdAsync(userId)`;
- `IsInRoleAsync`.

Pero no soporta estado activo/inactivo ni asignacion inicial de rol desde un caso de uso. La implementacion puede extender `IIdentityService` o crear un servicio especifico de administracion en Infrastructure. Evitar que Application dependa directamente de `ApplicationUser` si se quiere mantener Clean Architecture estricta.

### Decision recomendada para Clean Architecture

Preferencia: extender `IIdentityService` con metodos orientados a administracion:

```csharp
Task<UsersPageResult> GetUsersAsync(...);
Task<(Result Result, UserAdminDto? User)> CreateUserAsync(string email, string password, string role);
Task<Result> ActivateUserAsync(string userId);
Task<Result> DeactivateUserAsync(string userId, string? deactivatedBy);
Task<Result> DeleteUserAsync(string userId);
```

Si se agregan DTOs a Application, mantenerlos en Application y mapear desde Infrastructure. No exponer `ApplicationUser` fuera de Infrastructure.

### Seguridad y roles

Todos los endpoints de esta story deben requerir `Roles.Admin` en server-side. La UI tambien debe tener `[Authorize(Roles = Roles.Admin)]`, pero eso es solo ergonomia; la seguridad real vive en el API.

Roles validos existentes:

```csharp
Roles.Admin
Roles.Gerente
Roles.Supervisor
Roles.Empleado
```

No aceptar strings arbitrarios desde el frontend.

### Eliminacion segura

Aunque `RegistrosHoras.UserId` es string y no parece tener FK directa a `AspNetUsers`, borrar usuarios con historia destruye contexto operativo visible. Para esta story:

- Si hay `RegistrosHoras` o `SolicitudesExcepcion` asociadas, **no borrar fisicamente**; desactivar.
- Si no hay historia, se puede llamar a `UserManager.DeleteAsync`.
- Nunca borrar `RegistrosHoras`, `SolicitudesExcepcion` ni refresh tokens historicos desde esta accion. Si se quiere revocar refresh tokens al desactivar, hacerlo limpiamente en handler y cubrir con test; no es obligatorio para esta story si `ValidateCredentialsByIdAsync` ya bloquea refresh.

### Login/refresh de usuarios inactivos

`LoginCommandHandler` y `RefreshTokenCommandHandler` usan `IdentityService.ValidateCredentials*`. Por eso bloquear usuarios inactivos en esos metodos cubre login y refresh sin tocar handlers de Auth.

### UI esperada

Seguir el estilo de `SolicitudesExcepcionAdminPage.razor`:

- `MudStack` como layout principal;
- titulo con Poppins;
- `MudProgressLinear` para loading;
- `MudAlert` para error/success/empty state;
- `MudDataGrid` denso;
- `MudMessageBox` para confirmaciones;
- acciones con `MudIconButton` y `aria-label`.

No construir una landing page de administracion. La primera pantalla debe ser la herramienta usable de usuarios.

### Paginacion

La arquitectura documenta formato paginado:

```json
{
  "items": [],
  "totalCount": 0,
  "pageNumber": 1,
  "pageSize": 25
}
```

Para V1 puede implementarse paginacion server-side simple. Si el equipo prefiere usar todos los items con `MudDataGrid` local por bajo volumen inicial (30 colaboradores), documentar la decision, pero mantener contrato paginado en backend para crecer.
