# Story 6.1: Registrar Eventos Sensibles en Bitácora

Status: done

## Story

As a organización,
I want que las acciones sensibles queden registradas en una bitácora append-only,
So that exista trazabilidad confiable para auditorías.

## Acceptance Criteria

1. **Given** una acción sensible del sistema, **when** ocurre inicio de sesión exitoso, cambio de rol, alta/baja/reactivación/eliminación de usuario, modificación de descripción de registro, eliminación de registro autorizada, aprobación/rechazo de excepción o notificación enviada, **then** el sistema registra un evento en la tabla `BitacoraAuditoria` **and** incluye tipo de evento, actor (userId + email), entidad afectada, entidadId, timestamp UTC y metadata JSON relevante.

2. **Given** un evento de negocio crítico, **when** el handler correspondiente termina la operación sin error, **then** llama explícitamente a `IBitacoraService.RegistrarAsync` **and** el registro persiste en base de datos como operación append-only (sin posibilidad de UPDATE/DELETE desde la aplicación).

3. **Given** un fallo al registrar en bitácora, **when** `IBitacoraService.RegistrarAsync` lanza excepción, **then** no expone información sensible al usuario final **and** el error técnico queda en logs de Serilog.

4. **Given** la tabla `BitacoraAuditoria`, **when** se consulta directamente en DB, **then** no existen endpoints de escritura ni rutas de actualización ni eliminación en la API.

## Dev Notes

### Decisión de diseño: `BaseEntity` en lugar de `BaseAuditableEntity`

`BitacoraAuditoria` NO extiende `BaseAuditableEntity`. Si lo hiciera, el `AuditableEntityInterceptor` sobreescribiría los timestamps en cada SaveChanges, lo cual sería incorrecto para una entidad de auditoría. El timestamp lo setea el factory method `Crear()` usando `IClock.UtcNow` pasado por `BitacoraService`.

### Decisión de diseño: IBitacoraService guarda inmediatamente

`BitacoraService.RegistrarAsync` llama `SaveChangesAsync` internamente. Esto produce 2 operaciones de BD por handler (una para el negocio, una para la bitácora), pero es la opción más simple y robusta para V1. Alternativa futura: usar outbox pattern o domain events.

### Decisión de diseño: no propaga excepciones

`BitacoraService` captura internamente toda excepción y la loguea con Serilog. **La bitácora NUNCA debe romper el flujo de negocio.** Si el registro en bitácora falla, la operación de negocio ya fue exitosa y persistida.
