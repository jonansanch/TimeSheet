# Story 2.7: Accesibilidad y Estados del Formulario de Registro

Status: done

## Story

As a empleado,
I want completar el formulario usando teclado y recibir errores claros,
So that pueda registrar horas sin fricción ni ambigüedad.

## Acceptance Criteria

1. Dado el formulario de registro, cuando el usuario navega con `Tab`, `Shift+Tab`, `Enter`, `Space` y flechas, entonces puede completar cliente, proyecto, modalidad, horas y acciones sin mouse, y el foco visible nunca se pierde.
2. Dado un campo inválido, cuando el usuario intenta guardar, entonces aparece un mensaje específico bajo el input, y el mensaje indica qué corregir y está asociado con `aria-describedby`.
3. Dado una llamada de guardado en curso, cuando el usuario presiona Guardar, entonces el botón muestra spinner inline y texto "Guardando...", y el formulario conserva los valores si ocurre un error del sistema.

**POR ESTE bloque nuevo:**
```razor
<MudItem xs="12" md="6">
    <MudText id="modalidad-label-@turnoId" Typo="Typo.body2" Class="mb-1">Modalidad *</MudText>
    <div role="radiogroup"
         aria-labelledby="modalidad-label-@turnoId"
         aria-describedby="modalidad-error-@turnoId"
         style="display:flex;gap:6px;flex-wrap:wrap;">
        @foreach (var m in Modalidades)
        {
            var isSelected = model.Modalidad == m;
            var isFirst = m == Modalidades[0];
            var tabIdx = isSelected || (string.IsNullOrWhiteSpace(model.Modalidad) && isFirst) ? 0 : -1;
            <MudButton Variant="@(isSelected ? Variant.Filled : Variant.Outlined)"
                       Color="Color.Primary"
                       Size="Size.Small"
                       role="radio"
                       aria-checked="@(isSelected ? "true" : "false")"
                       tabindex="@tabIdx"
                       @onkeydown="@(e => HandleModalidadKey(e, model))"
                       OnClick="@(() => { model.Modalidad = m; _touchedModalidades.Add(turnoId); StateHasChanged(); })">
                @m
            </MudButton>
        }
    </div>
    <span id="modalidad-error-@turnoId" role="alert" aria-live="polite">
        @if (string.IsNullOrWhiteSpace(model.Modalidad) && _touchedModalidades.Contains(turnoId))
        {
            <MudText Typo="Typo.caption" Color="Color.Error">La modalidad es requerida.</MudText>
        }
    </span>
</MudItem>
```

**Roving tabindex:** Cuando no hay selección, el primer chip tiene `tabindex="0"`. Cuando hay selección, el chip seleccionado tiene `tabindex="0"` y el resto `tabindex="-1"`. Esto permite Tab al grupo y luego usar flechas para navegar.

### Implementación Exacta — HandleModalidadKey

Agregar en `@code`:
```csharp
private void HandleModalidadKey(KeyboardEventArgs e, ShiftFormModel model)
{
    if (e.Key is not ("ArrowLeft" or "ArrowRight" or "ArrowUp" or "ArrowDown"))
        return;
    var current = Array.IndexOf(Modalidades, model.Modalidad ?? Modalidades[0]);
    if (current < 0) current = 0;
    model.Modalidad = e.Key is "ArrowRight" or "ArrowDown"
        ? Modalidades[(current + 1) % Modalidades.Length]
        : Modalidades[(current - 1 + Modalidades.Length) % Modalidades.Length];
    StateHasChanged();
}
```

### Comportamiento Arrow Keys — Nota de Implementación

Con el patrón de roving tabindex en Blazor sin JSInterop:
- Usuario hace Tab al grupo → foco va al chip con `tabindex="0"` (seleccionado o primero)
- Usuario presiona ArrowRight/ArrowDown → `HandleModalidadKey` actualiza `model.Modalidad` → Blazor re-render → cambia qué chip tiene `tabindex="0"` visualmente (filled/outlined)
- El foco físico permanece en el chip original hasta el siguiente Tab (comportamiento aceptable sin JSInterop)
- Space/Enter en un chip también lo selecciona (MudButton responde nativamente)
- Esta implementación satisface AC1: el usuario PUEDE completar Modalidad sin mouse
