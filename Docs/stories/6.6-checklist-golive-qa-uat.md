# Story 6.6: Checklist de Go-Live, QA y UAT

Status: done

## Story

As a product owner,
I want una validación final antes de producción,
So that el equipo lance solo cuando los flujos críticos estén listos.

## Acceptance Criteria

1. **Given** el MVP implementado, **when** inicia preparación de go-live, **then** existe `Docs/operations/golive-checklist.md` con secciones: ambientes, accesos, roles, backups, monitoreo básico y ventana de reversión — y ningún defecto crítico de seguridad o flujo principal queda abierto.

2. **Given** QA funcional e integración, **when** se validan flujos J1–J4, **then** `Docs/operations/qa-script.md` cubre registro AM/PM, excepciones, reportes, administración y auditoría, con pasos ejecutables en Chrome, Edge y Firefox.

3. **Given** UAT de negocio, **when** el Product Owner (Jonathan) valida el flujo de registro, catálogos y reportes principales, **then** `Docs/operations/uat-checklist.md` existe con la firma/aprobación del PO y la salida a producción queda autorizada.

4. **Given** el primer día en producción, **when** se activa el sistema, **then** `Docs/operations/hypercare-plan.md` define las actividades de soporte activo y monitoreo para la semana 1.

  Checklist ejecutable predeployment con secciones:
  - **Infraestructura**: servidor IIS/Windows configurado, certificado SSL vigente, SQL Server accesible, carpeta de backups creada (`D:\Backups\Timesheet\`)

  - **Backups**: SQL Server Agent job `KPG_Timesheet_DailyBackup` creado y programado (23:00 diario), backup inicial manual ejecutado y verificado
  - **Monitoreo básico**: carpeta `logs/` accesible para el responsable técnico, alguien sabe cómo leer los logs
  - **Aplicación**: frontend Blazor WASM sirviendo correctamente, tiempo de carga inicial < 10s, login funciona
  - **Ventana de reversión**: backup pre-deploy guardado, procedimiento de rollback documentado en `backup-recovery.md` y conocido por el responsable
