# Story 2.1: Registrar Turno AM/PM con Validacion

Status: done

## Story

As a empleado,
I want registrar mis horas de entrada y salida por turno AM/PM,
so that mis horas diarias queden guardadas de forma precisa.

## Acceptance Criteria

1. Given un empleado autenticado, when abre la pantalla de registro, then la fecha de hoy aparece preseleccionada and el turno AM aparece activo por defecto.
2. Given el formulario de turno AM o PM, when el empleado ingresa hora de entrada, hora de salida, cliente, proyecto, modalidad, recurso, descripcion y lugar, then el sistema valida campos requeridos antes de guardar and no permite guardar si hay errores de validacion.
3. Given un formulario valido, when el empleado guarda el turno, then el registro queda asociado al usuario autenticado and el guardado responde en menos de 2 segundos en condiciones normales de red interna.

## Dev Notes

### Alcance

Esta historia implementa el primer flujo funcional de captura de horas. Debe entregar un registro AM/PM guardable y validado, pero no debe implementar todavia:

- Sugerencias recientes, filtros cliente-proyecto avanzados o chips accesibles de modalidad de Story 2.2.
- Banner de confirmacion, avance automatico AM a PM, "Copiar configuracion del AM" o "Jornada completa" de Story 2.3.
- Historial, eliminacion, retroactividad, excepciones, reportes, bitacora completa o notificaciones.

### Estado Actual Del Codigo

- Backend ya tiene solucion Clean Architecture en `Backend/KPG.Timesheet.sln` con `Api`, `Application`, `Domain`, `Infrastructure` y proyectos de test.
- Frontend oficial V1 es Blazor WebAssembly en `Fronted/src/WebUI/`; el Angular en `Fronted/legacy-angular-prototype` debe tratarse como prototipo legacy.
- Autenticacion base ya existe: Identity/JWT/refresh token en backend y `KpgAuthStateProvider` en WebUI.
- `IApplicationDbContext` actualmente solo expone `RefreshTokens`; esta historia debe extenderlo con `RegistrosHoras`.
- No existen todavia entidades de negocio `Empleado`, `Cliente`, `Proyecto`, `ParametroSistema` ni `BitacoraAuditoria`. Para esta historia, si se necesitan IDs de catalogo, usar campos/IDs simples sin adelantar CRUD de catalogos de Epic 4. Dejar nombres claros para reemplazo posterior.

### Reglas De Arquitectura

- Usar .NET 10, Clean Architecture, MediatR, FluentValidation, EF Core para escritura y MudBlazor en WebUI.
- Commands, handlers, validators y DTOs de resultado viven juntos en `Application/Features/RegistroHoras/Commands/CreateRegistroHoras/`.
- Endpoint REST plural y kebab-case: `/api/registros-horas`.
- Entidad: `RegistroHoras`; tabla: `RegistrosHoras`; DTO: `RegistroHorasDto`.
- Ningun endpoint debe envolver el handler en try/catch local. Los errores deben pasar por los handlers/middleware existentes y salir como Problem Details.
- La validacion backend es obligatoria aunque exista validacion frontend.
- El usuario asociado al registro viene del contexto autenticado del API; no confiar en un valor enviado por el cliente.

### Reglas UX

- La pagina de registro debe abrir lista para uso: fecha de hoy preseleccionada y turno AM activo.
- La experiencia AM/PM debe representarse como bloques o tabs, no como un unico formulario lineal enorme.
- Campos esperados en esta historia: fecha, turno, hora entrada, hora salida, cliente, proyecto, modalidad, recurso, descripcion y lugar.
- Validar en `onBlur` o al salir del campo; evitar mostrar errores mientras el usuario escribe.
- Boton Guardar deshabilitado hasta que los requeridos sean validos.
- Errores inline bajo cada campo, asociados al input cuando sea viable.
- Usar MudBlazor (`MudForm`, `MudTabs`, `MudTimePicker`, `MudTextField`, `MudButton`) y respetar tokens visuales KPG existentes.

### Datos Y Validaciones Minimas

- Fecha requerida; default: fecha local de hoy.
- Turno requerido; default: AM.
- Hora entrada requerida.
- Hora salida requerida y mayor que hora entrada.
- Cliente requerido.
- Proyecto requerido.
- Modalidad requerida.
- Recurso requerido.
- Descripcion requerida.
- Lugar requerido para cumplir AC de esta historia, aunque UX lo mencione como opcional en otra seccion; si negocio decide hacerlo opcional, actualizar esta story antes de dev.
- Debe impedirse guardar dos veces el mismo usuario/fecha/turno si el modelo ya permite verificar duplicados dentro del alcance. Si requiere infraestructura adicional, dejar test pendiente documentado para Story 2.3 o 3.6.
