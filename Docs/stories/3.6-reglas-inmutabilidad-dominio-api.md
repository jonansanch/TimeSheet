# Story 3.6: Reglas de Inmutabilidad en Dominio y API

Status: done

## Story

As a organizacion,
I want que los registros guardados sean inmutables salvo excepciones autorizadas,
so that la informacion que soporta facturacion sea confiable.

## Acceptance Criteria

1. **Given** un registro de horas ya guardado, **when** cualquier usuario intenta cambiar campos distintos a `Descripcion` mediante API o handler de aplicacion, **then** el sistema rechaza la modificacion antes de persistirla.

2. **Given** un registro de horas ya guardado, **when** Supervisor o Admin actualiza solo `Descripcion` usando `PATCH /api/registros-horas/{id}/descripcion`, **then** el cambio se permite y se conservan intactos `UserId`, `FechaRegistro`, `Turno`, `HoraEntrada`, `HoraSalida`, `Cliente`, `Proyecto`, `Modalidad`, `Recurso`, `Lugar` y `EsRetroactivo`.

3. **Given** una `SolicitudExcepcion` aprobada por Admin, **when** el usuario registra horas para la fecha autorizada, **then** el sistema permite crear el registro asociado a esa fecha y no habilita modificaciones generales sobre registros existentes.

4. **Given** una regla de negocio de dominio falla, **when** el API responde, **then** usa una excepcion de dominio tipada y el middleware la traduce a Problem Details sin exponer detalles sensibles.

5. **Given** una entidad de dominio valida sus invariantes, **when** la validacion falla por datos invalidos de negocio, **then** lanza la excepcion tipada de dominio en lugar de depender de `ArgumentException` o `InvalidOperationException` para reglas esperadas.

## Dev Notes

### Contexto de negocio

FR15 exige inmutabilidad post-guardado. La razon es financiera: `FechaRegistro`, `Turno`, horas, cliente, proyecto, modalidad, recurso, lugar y retroactividad soportan reportes/facturacion. La unica correccion permitida en Epic 3 es textual: `Descripcion`, implementada en Story 3.4 para Supervisor/Admin.

### Estado actual que se debe preservar

`RegistroHoras` ya tiene setters privados y solo expone `UpdateDescripcion(string nuevaDescripcion)`. Eso protege el modelo desde codigo normal, pero no basta como guardrail final porque EF puede marcar propiedades modificadas y futuros handlers podrian manipular el tracker. Esta story agrega una defensa en persistencia.

`PATCH /api/registros-horas/{id}/descripcion` ya existe y esta restringido a Supervisor/Admin en `RegistroHoras.cs` endpoint. No crear un endpoint de actualizacion general.

`CreateRegistroHorasCommandHandler` ya permite crear registros fuera de ventana cuando existe una `SolicitudExcepcion` aprobada para el mismo `UserId` y `FechaRegistro`. Esta story debe preservar ese comportamiento y dejar claro que la excepcion aprobada solo autoriza creacion, no edicion posterior.

### Patron recomendado para el interceptor

Usar EF Core ChangeTracker, no reflexion general sobre la entidad. La revision debe operar sobre propiedades modificadas:

```csharp
private static readonly HashSet<string> AllowedModifiedProperties =
[
    nameof(RegistroHoras.Descripcion),
    nameof(BaseAuditableEntity.LastModified),
    nameof(BaseAuditableEntity.LastModifiedBy)
];

var hasForbiddenChange = entry.Properties
    .Any(p => p.IsModified && !AllowedModifiedProperties.Contains(p.Metadata.Name));
```

Si `Created` o `CreatedBy` aparecen como modificados en una entidad existente, deben rechazarse. Son parte de la evidencia de auditoria inicial.

### Orden con AuditableEntityInterceptor

`AuditableEntityInterceptor` actualiza `LastModified` y `LastModifiedBy` en entidades modificadas. Por eso esos dos campos deben estar en la allowlist. El test de "solo descripcion" debe pasar con los interceptores activos para evitar falsos positivos.

### Excepcion de dominio vs ValidationException

No reemplazar FluentValidation ni `Application.Common.Exceptions.ValidationException`. Esa excepcion sigue siendo para validaciones de request/command. `DomainRuleException` es para invariantes del dominio que pueden fallar incluso si el request paso validacion.

### Campos inmutables de RegistroHoras

Lista cerrada que no debe cambiar despues de persistir:

| Campo | Motivo |
|---|---|
| `UserId` | Identidad del propietario del registro |
| `FechaRegistro` | Base del periodo facturable |
| `Turno` | Segmento AM/PM registrado |
| `HoraEntrada` / `HoraSalida` | Duracion facturable |
| `Cliente` / `Proyecto` | Imputacion financiera |
| `Modalidad` / `Recurso` / `Lugar` | Contexto operativo/reportable |
| `EsRetroactivo` | Evidencia de registro tardio |
| `Created` / `CreatedBy` | Auditoria de creacion |

`Descripcion` es el unico campo funcionalmente editable en esta epic.
