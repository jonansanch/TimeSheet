# Story 3.5: Eliminar Registros por Rol Autorizado

Status: done

## Story

As a supervisor or admin,
I want eliminar registros dentro de mi alcance,
so that pueda corregir errores operativos sin exponer datos fuera de mis permisos.

## Acceptance Criteria

1. **Given** un Supervisor o Admin autenticado, **when** envía `DELETE /api/registros-horas/{id}` para un registro que NO le pertenece, **then** el sistema lo elimina y retorna 204 No Content (antes retornaba 403).

2. **Given** un Empleado autenticado, **when** envía `DELETE /api/registros-horas/{id}` para un registro que NO le pertenece, **then** el sistema retorna 403 Forbidden (comportamiento previo conservado).

3. **Given** cualquier rol autenticado, **when** el registro solicitado no existe, **then** el sistema retorna 404 Not Found.

4. **Given** un Supervisor o Admin viendo la página Mis Registros, **when** hace clic en eliminar un registro propio, **then** el modal de confirmación muestra fecha, turno y el botón destructivo claramente diferenciado (comportamiento ya existente — se verifica que esté implementado correctamente).

5. **Given** la eliminación de un registro ajeno (desde API), **when** el handler procesa la solicitud, **then** no se requiere restricción de equipo (la restricción de equipo se aplica cuando exista entidad Team — anotado como deuda técnica).

## Dev Notes

### Cambio central: handler de eliminación

El único archivo de backend que cambia es `DeleteRegistroHorasCommandHandler.cs`. El cambio es quirúrgico:
- Se agrega la comprobación de rol (`IUser.Roles`) antes de verificar ownership
- Si el usuario es Supervisor o Admin, el ownership check se omite
- Si el usuario es Empleado (u otro rol sin privilegio), se mantiene el comportamiento actual

### IUser.Roles

`IUser` está en `Backend/src/Application/Common/Interfaces/IUser.cs`:
```csharp
public interface IUser
{
    string? Id { get; }
    List<string>? Roles { get; }
}
```

Usar con null-safety: `_user.Roles?.Contains(Roles.Admin) == true`. **No asumir** que `Roles` es siempre no-nulo.

### Restricción de "equipo" del Supervisor

Al igual que en Story 3.4, no existe entidad `Team` ni relación supervisor-empleado en la BD. Por eso el AC dice "rechaza registros fuera de su equipo" — pero esto no se puede implementar hoy. La simplificación es: cualquier Supervisor puede eliminar cualquier registro. Cuando exista el modelo de equipo se agregará la restricción. Documentar como deuda técnica en el handler con un comentario breve.

### Patrón TestUser para integration tests

En los tests de integración existentes (`CreateRegistroHorasCommandHandlerTests.cs`) se usa:
```csharp
private record TestUser(string Id) : IUser
{
    public List<string>? Roles => null;
}
```
Para los nuevos tests necesitamos pasar roles. Crear `TestUserWithRoles`:
```csharp
private record TestUserWithRoles(string Id, List<string> UserRoles) : IUser
{
    public List<string>? Roles => UserRoles;
}
```
O bien extender el `TestUser` existente añadiendo un segundo constructor. Verificar si `TestUser` ya es reutilizable desde otro archivo (probablemente está inline en el test file).

### Namespace conflict en test

El namespace `KPG.Timesheet.Infrastructure.IntegrationTests.RegistroHoras` hace que `RegistroHoras` sea ambiguo (namespace vs tipo). Usar alias al inicio del archivo:
```csharp
using RegistroHoras = KPG.Timesheet.Domain.Entities.RegistroHoras;
```
Y referencia con nombre completo en los métodos helper:
```csharp
private static KPG.Timesheet.Domain.Entities.RegistroHoras CreateRegistro(...) => new(...);
```

### Endpoint — NO se modifica

El endpoint `DELETE /api/registros-horas/{id}` ya existe con `auth = Empleado+Supervisor+Gerente+Admin`. No se crea un nuevo endpoint. No se agrega `[Authorize]` al `DeleteRegistroHorasCommand`. El control de acceso granular está en el handler.
