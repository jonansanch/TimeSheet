# Story 4.4: Gestionar Clientes y Proyectos

Status: review

## Story

As an admin,
I want mantener clientes y proyectos asociados,
so that los registros de horas reflejen correctamente la jerarquia de facturacion y el formulario muestre solo opciones vigentes.

## Acceptance Criteria

1. **Given** un Admin autenticado, **when** abre `/admin/clientes`, **then** ve una tabla con todos los clientes (activos e inactivos), su nombre, estado y acciones disponibles.

2. **Given** la tabla de clientes, **when** se renderiza, **then** soporta busqueda por nombre y paginacion sin recargar la pagina completa.

3. **Given** datos validos de nuevo cliente, **when** el Admin lo crea, **then** el sistema persiste el cliente con `Activo = true` y lo muestra en la tabla.

4. **Given** datos invalidos de nuevo cliente (nombre vacio o mayor a 200 caracteres), **when** el Admin intenta guardar, **then** el API responde con Problem Details/Validation y la UI muestra errores por campo sin cerrar el formulario.

5. **Given** un cliente activo, **when** el Admin lo desactiva, **then** el cliente pasa a `Activo = false` y desaparece del autocomplete de cliente en el formulario de registro. Sus proyectos desaparecen tambien del autocomplete aunque esten activos.

6. **Given** un cliente existente, **when** el Admin abre la gestion de sus proyectos, **then** ve una lista de proyectos vinculados a ese cliente con nombre, estado y acciones.

7. **Given** datos validos de nuevo proyecto, **when** el Admin lo crea bajo un cliente, **then** el sistema persiste el proyecto vinculado a ese cliente con `Activo = true`.

8. **Given** un proyecto activo, **when** el Admin lo desactiva, **then** el proyecto pasa a `Activo = false` y no aparece en el autocomplete de proyecto del formulario de registro para ese cliente. Los registros historicos con ese proyecto se conservan intactos.

9. **Given** un proyecto inactivo, **when** el Admin lo activa, **then** vuelve a aparecer en el autocomplete del formulario de registro.

10. **Given** el formulario de registro de horas (`KpgShiftForm`), **when** el usuario escribe en el autocomplete de Cliente, **then** las sugerencias incluyen unicamente los clientes activos del catalogo (mas los de historial reciente). **When** selecciona un cliente, **then** el autocomplete de Proyecto muestra solo los proyectos activos de ese cliente (mas los de historial reciente para ese cliente).

## Dev Notes

### Jerarquia Cliente → Proyecto

`Proyecto.ClienteId` es una FK a `Cliente.Id`. Un cliente puede tener N proyectos; un proyecto pertenece a exactamente un cliente.

La FK usa `DeleteBehavior.Restrict` para impedir eliminar un cliente que tenga proyectos, aunque en esta epic no hay eliminacion fisica — solo desactivacion. Esto protege la integridad referencial ante operaciones futuras.

`RegistroHoras.Cliente` y `RegistroHoras.Proyecto` siguen siendo strings. El catalogo alimenta el autocomplete; los registros historicos conservan el string original independientemente del estado del catalogo.

### Endpoint de catalogo para el formulario

`GET /api/clientes/catalogo` devuelve `List<ClienteConProyectosDto>` con clientes activos y sus proyectos activos anidados. Esto permite una sola llamada HTTP en `OnInitializedAsync` del formulario en lugar de dos llamadas (una para clientes, otra para proyectos).

Estructura de respuesta del catalogo:
```json
[
  {
    "id": 1,
    "nombre": "KPG",
    "proyectosActivos": ["Timesheet", "Nuevo Proyecto"]
  }
]
```

### Filtrado de proyectos en el formulario

`SearchProyectosAsync` ya tiene logica para filtrar por `model.Cliente`. La nueva implementacion debe:
1. Buscar en `_catalogoClientes` la entrada cuyo `Nombre` coincida (case-insensitive) con `model.Cliente`.
2. Si existe, usar `entry.ProyectosActivos` como base.
3. Si no existe en catalogo (cliente historico no cargado), usar sugerencias recientes como fallback.
4. En ambos casos, combinar con sugerencias recientes del mismo cliente y deduplicar.

### Duplicados de nombre

- `Cliente`: indice unico en `Nombre`. Error capturado en handler, convertido a `ValidationException("Ya existe un cliente con ese nombre.")`.
- `Proyecto`: indice unico compuesto `(ClienteId, Nombre)`. El mismo nombre puede existir en clientes diferentes. Error: `ValidationException("Ya existe un proyecto con ese nombre para este cliente.")`.

### Desactivacion en cascada

Desactivar un cliente NO desactiva automaticamente sus proyectos en la BD. Pero el endpoint `/api/clientes/catalogo` solo devuelve clientes activos, entonces si el cliente esta inactivo, ninguno de sus proyectos aparece en el formulario aunque tecnicamente esten activos en la BD. Este comportamiento satisface el AC 5.

### Organizacion de endpoints

Por claridad, se crean dos archivos de endpoints:
- `Clientes.cs` → todo bajo `/api/clientes` (CRUD clientes + proyectos sub-recurso).
- `Proyectos.cs` → `/api/proyectos/{id}` para `PUT` y toggle (operaciones sobre un proyecto por id sin necesitar el clienteId).

Alternativamente, todo en `Clientes.cs` es igualmente valido. Elegir lo mas limpio.
