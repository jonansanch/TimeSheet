# Story 3.4: Editar Solo Descripción de Registros Permitidos

Status: review

## Story

As a supervisor or admin,
I want editar únicamente la descripción de registros permitidos,
so that pueda corregir contexto textual sin alterar datos financieros críticos.

## Acceptance Criteria

1. **Given** un Supervisor o Admin autenticado, **when** envía PATCH `/api/registros-horas/{id}/descripcion` con una descripción válida, **then** el sistema actualiza solo el campo `Descripcion` del registro y retorna 204 No Content.

2. **Given** un Supervisor o Admin, **when** intenta modificar cualquier campo distinto a `Descripcion` (fecha, horas, cliente, proyecto, modalidad, recurso), **then** el endpoint no lo permite — el PATCH solo acepta `Descripcion`.

3. **Given** un usuario con rol Empleado (sin permiso de edición), **when** intenta llamar al endpoint PATCH, **then** el sistema retorna 403 Forbidden con Problem Details.

4. **Given** un Supervisor o Admin autenticado, **when** el registro solicitado no existe, **then** el sistema retorna 404 Not Found con Problem Details.

5. **Given** un Supervisor o Admin en la página Mis Registros, **when** hace clic en el ícono Editar de un registro, **then** se abre un diálogo con un campo de texto pre-cargado con la descripción actual, y puede modificarla y confirmar.

6. **Given** el diálogo de edición está abierto, **when** el usuario confirma con una descripción válida (no vacía, ≤ 1000 caracteres), **then** el frontend llama al PATCH, actualiza la fila en la tabla localmente sin recargar, y muestra mensaje de éxito.

7. **Given** el diálogo de edición, **when** el usuario intenta confirmar con descripción vacía o mayor a 1000 caracteres, **then** el botón Confirmar permanece deshabilitado (validación en cliente).

        if (string.IsNullOrWhiteSpace(_authState.AccessToken))
            return false;

        using var httpRequest = new HttpRequestMessage(HttpMethod.Patch, $"api/registros-horas/{id}/descripcion");
        httpRequest.Headers.Authorization = new AuthenticationHeaderValue("Bearer", _authState.AccessToken);
        httpRequest.Content = JsonContent.Create(new UpdateDescripcionRegistroRequest(descripcion));

## Dev Notes

### Arquitectura y patrones clave

- **Entidad inmutable con setters privados**: `RegistroHoras` expone todo como `{ get; private set; }`. La única forma de mutar campos es a través de métodos de dominio (como `UpdateDescripcion`). Esto es el patrón establecido — NO hacer `registro.Descripcion = value` directo desde el handler.
- **Alias para NotFoundException**: Hay colisión entre `Ardalis.GuardClauses.NotFoundException` y `KPG.Timesheet.Application.Common.Exceptions.NotFoundException`. Siempre usar el alias en los handlers:
  ```csharp
  using NotFoundException = KPG.Timesheet.Application.Common.Exceptions.NotFoundException;
  ```
- **Constructor de NotFoundException**: Solo acepta `(string)` o `(string, Exception)`. Usar: `throw new NotFoundException($"RegistroHoras con id '{request.RegistroId}' no fue encontrado.")`.
- **[Authorize] en commands**: Es `KPG.Timesheet.Application.Common.Security.AuthorizeAttribute` (no ASP.NET Core). Se resuelve vía `AuthorizationBehaviour<TRequest,TResponse>` en el pipeline MediatR.
- **IEndpointGroup pattern**: El nuevo método `UpdateDescripcion` se registra dentro del método `Map()` de la clase `RegistroHoras` (el endpoint group). No requiere ningún registro adicional — `MapEndpoints()` lo descubre por reflexión.
- **MudBlazor 8.x IMudDialogInstance**: `MudDialog.Close()` y `MudDialog.Cancel()` son **síncronos** (no CloseAsync / CancelAsync). El parámetro es `IMudDialogInstance`, no `MudDialogInstance`.
- **DialogParameters con genérico tipado**: Usar `DialogParameters<KpgEditDescripcionDialog>` para type-safe parameter binding.

### Restricción de "equipo" del Supervisor

Actualmente no existe una entidad `Team` ni relación supervisor-empleado. Por eso, el handler NO implementa restricción de ownership de equipo — cualquier Supervisor puede editar la descripción de cualquier registro. Esta es una simplificación consciente: la restricción de equipo se implementará cuando exista el modelo de equipo (futura épica). El `[Authorize(Roles = ...)]` protege contra roles sin permiso.

### Patrón del endpoint PATCH

El endpoint recibe un body JSON con solo `Descripcion`. No es PATCH parcial tipo JSON-Patch — es un PATCH simplificado que acepta el campo que puede modificarse. Esto es coherente con el enfoque del proyecto (no usa System.Text.Json JsonPatch).

### Roles en el frontend

`CurrentUserService.IsInRoleAsync(role)` consulta los claims del JWT. Los roles disponibles están en `KPG.Timesheet.WebUI.Shared.Constants.Roles` (ver cómo se usa en `NavMenu.razor`: `Roles.Supervisor`, `Roles.Admin`).

### Cómo HistorialRegistroResponse es un record

`HistorialRegistroResponse` es un `record` en `RegistroHorasModels.cs`. Para actualizar localmente la descripción en la lista usar `with`: `_registros[idx] = item with { Descripcion = nuevaDescripcion }`. Esto funciona igual que en `SolicitudesExcepcionAdminPage.razor` (precedente de la historia 3.3).
