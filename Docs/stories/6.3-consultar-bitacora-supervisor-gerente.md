# Story 6.3: Consultar Bitácora por Supervisor y Gerente

Status: done

## Story

As a supervisor or gerente,
I want consultar la bitácora según mi alcance autorizado,
So that pueda revisar acciones relevantes sin acceder a información no autorizada.

## Acceptance Criteria

1. **Given** un Gerente autenticado, **when** abre `/bitacora`, **then** ve la bitácora completa del sistema (todos los eventos, todos los actores) con los mismos filtros que el Admin (fecha, usuario, tipo de evento).

2. **Given** un Supervisor autenticado, **when** abre `/bitacora`, **then** solo ve eventos cuyo actor tiene rol Empleado o Supervisor **and** no puede ver eventos generados por Gerentes ni Admins.

3. **Given** un Supervisor o Gerente autenticado, **when** consulta `GET /api/bitacora/mi-alcance`, **then** el backend determina el scope según el rol del token JWT y filtra los resultados server-side (NFR9).

4. **Given** un usuario sin rol Supervisor ni Gerente, **when** llama a `GET /api/bitacora/mi-alcance`, **then** recibe `403 Forbidden`.

5. **Given** resultados de bitácora, **when** se muestran en tabla, **then** usan el mismo `MudDataGrid` compacto de Story 6.2 (timestamp, chip de tipo, actor, entidad, ID entidad, metadata con tooltip).

6. **Given** filtros sin resultados, **when** el usuario consulta, **then** se muestra estado vacío y los filtros se conservan.

7. **Given** un Admin autenticado, **when** navega al NavMenu, **then** solo ve "Bitácora" en la sección Admin (enlace a `/admin/bitacora`); el enlace `/bitacora` NO aparece para Admin.

- [x] Crear `Backend/src/Infrastructure/Bitacora/GetBitacoraAlcanceQueryHandler.cs`
  - Inyecta `IDbConnection db` y `IUser user` (ambos disponibles en DI)
  - Usa DOS bloques SQL constantes: `SqlCompleto` (Gerente) y `SqlEquipo` (Supervisor)
  - Lógica de selección:
    ```csharp
    bool isSupervisor = user.Roles?.Contains(Roles.Supervisor) == true
                        && user.Roles?.Contains(Roles.Gerente) != true;
    var sql = isSupervisor ? SqlEquipo : SqlCompleto;
    ```
  - `SqlCompleto` — idéntico al SQL de `GetBitacoraQueryHandler` (sin filtro de actor por rol):
    ```sql
    SELECT b.Id, b.TipoEvento, b.ActorId, b.ActorEmail,
           ISNULL(u.NombreCompleto, b.ActorEmail) AS ActorNombre,
           b.EntidadAfectada, b.EntidadId, b.Timestamp, b.MetadataJson
    FROM   BitacoraAuditoria b
    LEFT   JOIN AspNetUsers u ON u.Id = b.ActorId
    WHERE  (@Desde      IS NULL OR CAST(b.Timestamp AS date) >= @Desde)
      AND  (@Hasta      IS NULL OR CAST(b.Timestamp AS date) <= @Hasta)
      AND  (@ActorId    IS NULL OR b.ActorId = @ActorId)
      AND  (@TipoEvento IS NULL OR b.TipoEvento = @TipoEvento)
    ORDER  BY b.Timestamp DESC
    OFFSET 0 ROWS FETCH NEXT 500 ROWS ONLY
    ```
  - `SqlEquipo` — agrega subquery de scope para Supervisor:
    ```sql
    SELECT b.Id, b.TipoEvento, b.ActorId, b.ActorEmail,
           ISNULL(u.NombreCompleto, b.ActorEmail) AS ActorNombre,
           b.EntidadAfectada, b.EntidadId, b.Timestamp, b.MetadataJson
    FROM   BitacoraAuditoria b
    LEFT   JOIN AspNetUsers u ON u.Id = b.ActorId
    WHERE  b.ActorId IN (
               SELECT ur.UserId
               FROM   AspNetUserRoles ur
               JOIN   AspNetRoles r ON r.Id = ur.RoleId
               WHERE  r.Name IN ('Empleado', 'Supervisor')
           )
      AND  (@Desde      IS NULL OR CAST(b.Timestamp AS date) >= @Desde)
      AND  (@Hasta      IS NULL OR CAST(b.Timestamp AS date) <= @Hasta)
      AND  (@ActorId    IS NULL OR b.ActorId = @ActorId)
      AND  (@TipoEvento IS NULL OR b.TipoEvento = @TipoEvento)
    ORDER  BY b.Timestamp DESC
    OFFSET 0 ROWS FETCH NEXT 500 ROWS ONLY
    ```

### Lógica de scope: Gerente vs Supervisor

La determinación del scope se hace en el handler mediante:
```csharp
bool isSupervisor = user.Roles?.Contains(Roles.Supervisor) == true
                    && user.Roles?.Contains(Roles.Gerente) != true;
```

El doble chequeo protege contra el caso hipotético de un usuario con ambos roles (Supervisor y Gerente) — en ese caso se trata como Gerente (scope completo). Es el resultado más permisivo, que es el correcto.

### SqlEquipo: por qué filtrar por AspNetUserRoles

El sistema no tiene una tabla de asignación explícita Supervisor→Empleado. No hay "equipos" definidos. La interpretación más cercana a "su equipo" en una empresa plana de 30 personas es: "todos los que tienen rol Empleado o Supervisor" (excluyendo Gerente y Admin).

Esto significa que el Supervisor ve:
- Sus propias acciones (ActorId = el Supervisor mismo, que tiene rol Supervisor ✓)
- Las acciones de todos los Empleados

Y NO ve:
- Acciones de Admin (gestión de sistema, cambios de configuración)
- Acciones de Gerente

La subquery en `SqlEquipo`:
```sql
b.ActorId IN (
    SELECT ur.UserId
    FROM   AspNetUserRoles ur
    JOIN   AspNetRoles r ON r.Id = ur.RoleId
    WHERE  r.Name IN ('Empleado', 'Supervisor')
)
```
Los valores `'Empleado'` y `'Supervisor'` coinciden exactamente con las constantes de `Roles.Empleado` y `Roles.Supervisor` (ambas son strings `"Empleado"` y `"Supervisor"`).
