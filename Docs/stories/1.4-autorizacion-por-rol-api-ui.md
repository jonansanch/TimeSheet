# Story 1.4: Autorización por Rol en API y UI

Status: done

## Story

As a usuario de KPG,
I want ver y ejecutar solo las funcionalidades permitidas por mi rol,
So that la información y las acciones sensibles queden protegidas.

## Acceptance Criteria

1. **Given** un usuario con rol Empleado, Supervisor, Gerente o Admin, **when** solicita un endpoint protegido, **then** el API verifica el rol con autorización server-side, **and** rechaza accesos no permitidos aunque el frontend haya sido manipulado.

2. **Given** un usuario autenticado, **when** se renderiza la navegación principal, **then** solo aparecen las secciones permitidas para su rol, **and** las secciones no permitidas no se muestran deshabilitadas.

3. **Given** endpoints protegidos, **when** se revisan los endpoints de la Épica 1, **then** usan autorización por rol donde aplique, **and** los errores siguen formato Problem Details.

## Tasks / Subtasks

- [x] Definir constantes de roles en `Domain/Constants/Roles.cs`: Admin, Gerente, Supervisor, Empleado
- [x] Configurar autorización en `Program.cs` con políticas por rol

### Problem Details para errores de rol

- 401 (`UnauthorizedAccessException`): no autenticado o JWT inválido
- 403 (`ForbiddenAccessException`): autenticado pero sin permiso
- Ambos manejados por `ProblemDetailsExceptionHandler`

### NFR9 — Defensa en profundidad

La UI solo muestra lo permitido, pero cada endpoint verifica rol server-side independientemente. No hay endpoint que dependa solo de que el frontend haya ocultado la opción.

### NavMenu por rol

- Empleado: Inicio, Registro, Mis Registros
- Supervisor: + Dashboard, Reportes
- Gerente: + Dashboard gerencial, Reportes
- Admin: + todas las anteriores + sección Admin completa
