# Story 4.5: Gestionar Parámetros del Sistema

Status: done

## Story

As an admin,
I want administrar modalidades, lugares de trabajo y parámetros de configuración del sistema (ventana retroactiva, umbral de notificaciones),
so that las reglas operativas puedan ajustarse sin cambios de código y el formulario de registro muestre solo opciones vigentes.

## Acceptance Criteria

1. **Given** un Admin autenticado, **when** accede a Parámetros desde el menú lateral, **then** ve un grupo colapsable con tres subpáginas independientes: (a) `/admin/parametros/configuracion` — Configuración General (ventana retroactiva, umbral de notificaciones), (b) `/admin/parametros/modalidades` — Modalidades, (c) `/admin/parametros/lugares` — Lugares de Trabajo.

2. **Given** la sección de modalidades, **when** se renderiza, **then** muestra tabla con todas las modalidades (activas e inactivas), nombre, estado y acciones (editar, toggle).

3. **Given** datos válidos de nueva modalidad, **when** el Admin la crea, **then** el sistema persiste la modalidad con `Activo = true` y la muestra en la tabla.

4. **Given** datos inválidos (nombre vacío o mayor a 100 caracteres), **when** el Admin intenta guardar, **then** el API responde con Problem Details/Validation y la UI muestra errores por campo sin cerrar el formulario.

5. **Given** una modalidad activa, **when** el Admin la desactiva, **then** la modalidad pasa a `Activo = false` y desaparece como chip en el formulario de registro.

6. **Given** la sección de lugares de trabajo, **when** se renderiza, **then** muestra tabla con todos los lugares (activos e inactivos), nombre, estado y acciones (editar, toggle). Mismo patrón que modalidades.

7. **Given** el campo de ventana retroactiva en la sección de configuración, **when** el Admin modifica el valor y guarda, **then** el nuevo valor queda en `ParametrosSistema` con clave `VentanaRetroactividad` y la lógica de registro usa el nuevo valor.

8. **Given** el campo de umbral de notificaciones, **when** el Admin modifica el valor y guarda, **then** el nuevo valor queda en `ParametrosSistema` con clave `DiasUmbralNotificacion`. El valor debe ser entero positivo entre 1 y 30.

9. **Given** el formulario de registro de horas (`KpgShiftForm`), **when** carga, **then** las modalidades disponibles como chips provienen de `GET /api/modalidades/activas` (no hardcodeadas). Si la carga falla, usa fallback `["Presencial", "Remoto", "Hibrido"]`.

10. **Given** el formulario de registro de horas, **when** carga, **then** el dropdown de Lugar proviene de `GET /api/lugares-trabajo/activos` (no hardcodeado). Si la carga falla, usa fallback `["Presencial Oficina", "Presencial Viaje", "Presencial Cliente", "Remoto"]`.

11. **Given** una modalidad o lugar desactivado, **when** un empleado abre el formulario de registro, **then** ese valor no aparece como opción para nuevas capturas. Los registros históricos conservan el string original.

Todas las entidades de catálogo del proyecto (`Empleado`, `Cliente`, `Proyecto`) siguen el mismo patrón:

### `HandleModalidadKey` depende de Modalidades siendo un array

```csharp
private void HandleModalidadKey(KeyboardEventArgs e, ShiftFormModel model)
{
    if (e.Key is not ("ArrowLeft" or "ArrowRight" or "ArrowUp" or "ArrowDown"))
        return;
    var current = Array.IndexOf(Modalidades, model.Modalidad ?? Modalidades[0]);
    // ...
    model.Modalidad = ... Modalidades[(current + 1) % Modalidades.Length] ...
}
```

Al cambiar a `_modalidades: List<string>`, usar:
```csharp
var lista = _modalidades.Count > 0 ? _modalidades : (List<string>)["Presencial", "Remoto", "Hibrido"];
var current = lista.IndexOf(model.Modalidad ?? lista[0]);
if (current < 0) current = 0;
model.Modalidad = e.Key is "ArrowRight" or "ArrowDown"
    ? lista[(current + 1) % lista.Count]
    : lista[(current - 1 + lista.Count) % lista.Count];
```

### ParametroSistema — clave `DiasUmbralNotificacion`

El parámetro `DiasUmbralNotificacion` se usa en Stories 5.9 y 5.10. En esta story solo se implementa la configuración Admin. El valor default es `3` (días sin registro antes de notificar). Clave constante en `Domain/Constants/ParametrosSistema.cs`.

### Lógica de `UpdateVentanaRetroactividad` en backend

El parámetro puede existir o no (ya existe en seed). El handler debe:
1. `FirstOrDefault(p => p.Clave == ...)`.
2. Si existe: `param.Valor = request.Dias.ToString();`.
3. Si no existe: `_context.ParametrosSistema.Add(new ParametroSistema { Clave = ..., Valor = ... })`.
4. `await _context.SaveChangesAsync(ct)`.

**IMPORTANTE:** `ParametroSistema` tiene setters públicos (no privados como las otras entidades). Ver `Backend/src/Domain/Entities/ParametroSistema.cs` — `Clave` y `Valor` son `public string Clave { get; set; }`. Esto es correcto para un parámetro genérico de sistema.
