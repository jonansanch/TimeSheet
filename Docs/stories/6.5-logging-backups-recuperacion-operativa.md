# Story 6.5: Logging, Backups y Recuperación Operativa

Status: done

## Story

As a responsable técnico,
I want tener logging estructurado, backups automáticos y procedimientos de recuperación documentados,
So that la plataforma sea operable en producción y se pueda diagnosticar cualquier fallo.

## Acceptance Criteria

1. **Given** ambiente de producción, **when** la aplicación procesa cualquier request o genera un error, **then** Serilog escribe logs estructurados a archivo con rotación diaria, retención de 30 días, y nivel mínimo Information.

2. **Given** los logs generados, **when** se revisa el archivo, **then** no contienen la JWT Key, la contraseña de SMTP, ni el password del connection string. Los logs de comandos EF Core (que podría incluir valores de parámetros) están suprimidos en producción.

3. **Given** un error 500 no controlado, **when** llega al `ProblemDetailsExceptionHandler`, **then** la excepción completa (con stack trace) queda registrada en los logs via `ILogger` antes de devolver la respuesta genérica al cliente.

4. **Given** ambiente de producción de SQL Server, **when** termina cada día operativo, **then** existe un T-SQL de SQL Server Agent documentado que ejecuta respaldo `BACKUP DATABASE` al archivo con timestamp, con verificación `RESTORE VERIFYONLY`.

5. **Given** un incidente que requiera recuperación, **when** se sigue el procedimiento documentado, **then** la recuperación objetivo es ≤ 4 horas siguiendo los pasos en `Docs/operations/backup-recovery.md`.

6. **Given** horario laboral lunes a viernes 7am-8pm, **when** se define el SLA, **then** la meta de disponibilidad es ≥ 99% documentada en `Docs/operations/availability-sla.md`.

## Dev Notes

### Serilog NO está instalado — esta es la instalación inicial

El proyecto usa el logging predeterminado de ASP.NET Core (ILogger de Microsoft, sin providers de archivo). Esta historia instala Serilog desde cero.

- **Versiones**: `Serilog.AspNetCore` 9.0.0 (compatible con .NET 9/10), `Serilog.Sinks.File` 6.0.0
- El proyecto usa **Central Package Management** (`Directory.Packages.props` en `Backend/`), entonces:
  1. Las versiones se declaran SOLO en `Directory.Packages.props` (con `<PackageVersion>`)
  2. Los proyectos referencian SIN versión (con `<PackageReference>`)
  3. NO poner versiones en los `.csproj`

### ReadFrom.Configuration — patrón estándar

`lc.ReadFrom.Configuration(ctx.Configuration)` hace que Serilog lea toda su configuración de `appsettings.json` (sección `"Serilog"`). Con este patrón:
- No hay configuración hardcoded en código
- `appsettings.Development.json` puede sobrescribir niveles
- `appsettings.Production.json` puede eliminar el sink Console

**Importante**: La sección se llama `"Serilog"` (no `"Logging"`). Ambas pueden coexistir, pero como ya no usamos el proveedor de Microsoft, la sección `"Logging"` en `appsettings.json` se elimina para evitar confusión.

### Secretos: por qué no se filtran con código

Los secretos (JWT Key, contraseña SMTP, password de BD) NUNCA deben llegar al logger por las siguientes razones:
1. El `ProblemDetailsExceptionHandler` devuelve solo `Title` genérico al cliente — no incluye Connection String ni JWT Key
2. Serilog loguea solo los mensajes que el código explícitamente construye — nunca objetos `IConfiguration`
3. `Microsoft.EntityFrameworkCore.Database.Command` se suprime en producción → evita logging de SQL con parámetros que podrían incluir valores sensibles
4. La excepción en el log `exception.Message` puede aparecer en raras excepciones de autenticación — por eso el `ProblemDetailsExceptionHandler` ya maneja `UnauthorizedAccessException` por separado (sin loguear) y solo loguea el `_` case (500)

No agregar `Serilog.Enrichers.Sensitive` ni destructuring policies — el riesgo de filtración ya está mitigado por diseño.

### app.UseSerilogRequestLogging — dónde colocarlo

Debe ir en el pipeline DESPUÉS de `UseHttpsRedirection` y ANTES de `UseAuthentication`. Esto asegura:
- Los requests que llegan por HTTP (y se redirigen a HTTPS) no se loguean dos veces
- La identidad del usuario autenticado NO está disponible en el log del request (esto es intencional — el log de request es a nivel infraestructura, no negocio)

### ProblemDetailsExceptionHandler — cambio de constructor

La clase actualmente no tiene constructor explícito. Al agregar `ILogger<ProblemDetailsExceptionHandler>` como dependencia, se usa **primary constructor de C#**:
```csharp
public class ProblemDetailsExceptionHandler(ILogger<ProblemDetailsExceptionHandler> logger) : IExceptionHandler
```
Esto es consistente con el patrón usado en los handlers Dapper del proyecto (e.g. `GetBitacoraQueryHandler(IDbConnection db)`).

El DI container de ASP.NET Core registra `IExceptionHandler` con `AddExceptionHandler<ProblemDetailsExceptionHandler>()` (en `DependencyInjection.cs`). La inyección de `ILogger<T>` es automática — no se requiere cambio en el registro.

### Tests de integración — no se rompen con Serilog

Los tests de integración usan `WebApplicationFactory<Program>` que respeta la configuración de `appsettings.json` + `appsettings.Development.json`. Con Serilog:
- `ReadFrom.Configuration` lee la configuración del `TestServer` — funciona igual
- Los tests no tienen `ILogger<T>` mockeado — Serilog simplemente escribe (o no escribe si el nivel está por encima)
- La carpeta `logs/` puede no existir en el directorio de tests — Serilog.Sinks.File crea el directorio automáticamente si no existe

Si los tests fallaran por el sink de archivo (permisos en CI), el fallback es configurar un nivel `Fatal` en `appsettings.Testing.json` o usar `MinimumLevel.Override` para los sinks. Pero en este proyecto no hay ese problema — los tests corren en el mismo equipo Windows.

### Documentos de operaciones — NO son código

Los archivos en `Docs/operations/` son documentación Markdown:
- `backup-recovery.md`: incluye el T-SQL de SQL Server Agent y los pasos manuales de recuperación
- `availability-sla.md`: define el SLA del sistema

Estos documentos NO requieren build ni tests. Son artefactos de operaciones para el equipo técnico de KPG.
