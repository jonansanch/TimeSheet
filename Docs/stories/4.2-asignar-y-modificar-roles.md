# Story 4.2: Asignar y Modificar Roles

Status: done

## Story

As an admin,
I want asignar y modificar roles de usuarios,
so that cada persona tenga permisos correctos segun su funcion.

## Acceptance Criteria

1. **Given** un Admin autenticado, **when** abre `/admin/usuarios`, **then** puede ver el rol actual de cada usuario y acceder a una accion para cambiarlo.

2. **Given** un usuario existente, **when** el Admin asigna rol `Empleado`, `Supervisor`, `Gerente` o `Admin`, **then** el sistema reemplaza el rol anterior por el rol seleccionado y devuelve el usuario actualizado.

3. **Given** un cambio de rol, **when** el Admin confirma la modificacion, **then** el backend valida que el rol pertenezca al catalogo permitido y rechaza valores fuera de `Roles.Admin`, `Roles.Gerente`, `Roles.Supervisor`, `Roles.Empleado`.

4. **Given** un usuario con sesion activa, **when** su rol cambia, **then** el nuevo rol se refleja en el siguiente refresh/relogin y el API sigue validando permisos server-side con los claims actualizados.

5. **Given** el Admin intenta cambiar su propio rol, **when** la solicitud llega al API, **then** el sistema rechaza la operacion para evitar dejar al sistema sin administrador operativo.

6. **Given** el Admin intenta quitar el rol `Admin` al ultimo administrador activo, **when** confirma el cambio, **then** el sistema rechaza la operacion para preservar al menos una cuenta administradora activa.

7. **Given** el cambio de rol falla por validacion o error de Identity, **when** la UI recibe la respuesta, **then** muestra un mensaje claro sin ocultar la tabla ni perder el estado actual.

8. **Given** un usuario inactivo, **when** el Admin modifica su rol, **then** el rol queda actualizado pero el usuario sigue inactivo hasta que sea activado explicitamente.

## Dev Notes

### Alcance exacto de Story 4.2

Esta story solo cubre asignacion/modificacion del rol Identity de usuarios existentes. No crea usuarios, no activa/desactiva cuentas, no elimina cuentas y no implementa empleados, clientes, proyectos ni parametros. Es una extension directa de Story 4.1.

La story debe reemplazar el rol principal del usuario. El sistema actual muestra un solo rol por usuario en `UserAdminDto.Role`; por consistencia, la UI y API deben tratar el rol como unico para V1.

### Contexto de Story 4.1

Story 4.1 ya dejo implementado:

- `ApplicationUser` con `IsActive`, `Created`, `DeactivatedAt`, `DeactivatedBy`.
- `IIdentityService` con listado, creacion con rol inicial, activacion, desactivacion y eliminacion.
- `IdentityService.ValidateCredentialsAsync` y `ValidateCredentialsByIdAsync` devuelven roles desde Identity.
- Endpoint `/api/users` Admin-only.
- UI `/admin/usuarios` con `UserAdminResponse.Role`.
- `UserAdminRepository.ReadErrorAsync` parsea Problem Details y ValidationProblemDetails.

No reimplementar esa base. Extenderla.

### Politica de sesion para cambios de rol

El acceptance de la Epic dice que el nuevo rol se aplica en el siguiente refresh/relogin segun politica de sesion. La politica actual del sistema es:

- El access token contiene roles al momento de emitirse.
- `RefreshTokenCommandHandler` llama `ValidateCredentialsByIdAsync`.
- `ValidateCredentialsByIdAsync` consulta roles actuales en Identity.

Por tanto, no es necesario revocar access tokens vigentes en esta story. El rol nuevo se reflejara cuando el usuario haga refresh o vuelva a iniciar sesion. El API debe seguir protegido server-side con roles en cada endpoint.

### Regla de seguridad: admins operativos

Story 4.1 ya rechazo desactivar/eliminar la propia cuenta. Esta story debe rechazar cambiar el propio rol por la misma razon. Adicionalmente, no debe permitir quitar el rol `Admin` al ultimo admin activo.

La verificacion de "ultimo Admin activo" debe considerar:

- usuarios con `IsActive == true`;
- usuarios que pertenecen a `Roles.Admin`;
- excluir al usuario objetivo cuando el nuevo rol no sea `Admin`.

Si no queda otro admin activo, retornar error claro: `Debe existir al menos un administrador activo.`

### Roles validos

Usar exclusivamente constantes existentes:

```csharp
Roles.Admin
Roles.Gerente
Roles.Supervisor
Roles.Empleado
```

No aceptar strings arbitrarios desde frontend ni backend. La validacion visible al usuario debe estar en Espanol.

### UI esperada

Seguir el estilo actual de `UsuariosAdminPage.razor`:

- `MudDataGrid` denso.
- Acciones con `MudIconButton`.
- `MudMessageBox` para acciones destructivas; cambio de rol puede usar dialogo con confirmacion explicita en el boton Guardar.
- Alertas inline para success/error.
- No agregar dashboard admin general en esta story.

El dialogo debe tener tamano medio consistente con los ajustes recientes: `MaxWidth.Small`, `FullWidth = true`.
