## Cambios implementados en esta sesion

### Optimizaciones de rendimiento (Fixes A-D)

#### Fix A — IdentityService.GetUsersAsync: 1 round-trip Dapper

**Problema:** `GetUsersAsync` ejecutaba 1 query por usuario para obtener su rol (N+1).

**Solucion:**
- `QueryMultipleAsync` con `COUNT(*) + ROW_NUMBER()` paginado — 1 solo round-trip Dapper via `GridReader`.
- Clase privada `UserAdminRow` (propiedades regulares) para que Dapper IL maneje `Int64→bool` de SQLite correctamente.
- `DapperTypeHandlers.cs`: nuevos `DateTimeOffsetTypeHandler` y `NullableDateTimeOffsetTypeHandler`.
- Handlers registrados en `DependencyInjection.cs` y en el setup de los integration tests.

**Archivos:**
- `Backend/src/Infrastructure/Identity/IdentityService.cs`
- `Backend/src/Infrastructure/Data/DapperTypeHandlers.cs`
- `Backend/src/Infrastructure/DependencyInjection.cs`
- `Backend/tests/Infrastructure.IntegrationTests/Users/IdentityServiceUserAdminTests.cs`

#### Fix B — SqlPendientes: LEFT JOIN acotado a 90 dias

**Problema:** `DashboardRepository.SqlPendientes` hacia full scan del historial completo de `RegistrosHoras`.

**Solucion:** Agregado `AND r.FechaRegistro >= @LimiteBusqueda` (hoy − 90 dias) al LEFT JOIN. Usa el indice `IX_RegistrosHoras_FechaRegistro`.

**Archivo:** `Backend/src/Infrastructure/Dashboard/DashboardRepository.cs`

#### Fix C — GetCatalogoClientesConProyectos: 1 LEFT JOIN

**Problema:** Handler ejecutaba 2 queries EF (clientes activos + proyectos activos) y hacia el join en C#.

**Solucion:** LINQ LEFT JOIN con `DefaultIfEmpty()` → 1 sola query SQL. GroupBy en memoria sobre dataset pequeño.

**Archivo:** `Backend/src/Application/Features/Catalogos/Clientes/Queries/GetCatalogoClientesConProyectos/GetCatalogoClientesConProyectosQueryHandler.cs`

#### Fix D — GetSolicitudesExcepcion: filtro de ventana 180 dias

**Problema:** `ToListAsync()` sin filtro cargaba todo el historial de solicitudes.

**Solucion:** Filtro `WHERE Estado = Pendiente OR FechaRegistro >= hoy − 180 dias`. Preserva solicitudes pendientes sin ventana.

**Archivo:** `Backend/src/Application/Features/SolicitudesExcepcion/Queries/GetSolicitudesExcepcion/GetSolicitudesExcepcionQueryHandler.cs`

### Bugs corregidos durante QA (Fixes E-G)

#### Fix E — SetFecha no actualizaba _dateAvailability

**Sintoma:** Usuario selecciona fecha fuera de ventana desde el mini-calendario → frontend no bloquea → backend lanza `ValidationException`.

**Causa:** `KpgShiftForm.SetFecha(fecha)` es llamado directamente por `RegistroPage` (no pasa por `HandleDateChanged`). Actualizaba `_fecha` pero no `_dateAvailability`, dejando el valor anterior del estado.

**Fix:** Recalculacion de `_dateAvailability` dentro de `SetFecha` usando `BusinessDayCalculator.GetAvailability`.

**Archivo:** `Fronted/src/WebUI/Features/Registro/Components/KpgShiftForm.razor`

#### Fix F — Campo Descripcion sin MaxLength

**Sintoma:** El campo de descripcion en el formulario de registro no limitaba la cantidad de caracteres en el frontend, lo que podia causar error de base de datos al superar los 1000 chars de `NVARCHAR(1000)`.

**Fix:** Agregado `MaxLength="1000"` y `Counter="1000"` al `MudTextField` de Descripcion.

**Archivo:** `Fronted/src/WebUI/Features/Registro/Components/KpgShiftForm.razor`

#### Fix G — GET /api/users bloqueado para Gerente

**Sintoma:** Gerente logueado en la pantalla de Bitacora no cargaba la lista de usuarios → `ForbiddenAccessException` en backend.

**Causa raiz — dos niveles de autorizacion:**
1. `Api/Endpoints/Users.cs` — `RequireAuthorization(adminOnly)` en el GET endpoint.
2. `Application/Features/Users/Queries/GetUsers/GetUsersQuery.cs` — `[Authorize(Roles = Roles.Admin)]` procesado por `AuthorizationBehavior` en MediatR.

Ambos requerian solo `Admin`. El primero bloqueaba la request, pero aunque se corrigiera solo ese, el segundo hubiera lanzado `ForbiddenAccessException` internamente.

**Fix:** Ambos actualizados a `Admin, Gerente, Supervisor`. Las operaciones de escritura (Create, Activate, Deactivate, ChangeRole, Delete) siguen siendo exclusivas de Admin.

**Archivos:**
- `Backend/src/Api/Endpoints/Users.cs`
- `Backend/src/Application/Features/Users/Queries/GetUsers/GetUsersQuery.cs`

## Patron importante: Autorizacion en dos niveles

En Clean Architecture, la autorizacion se aplica en dos capas independientes:

```
Request HTTP
    │
    ▼
Endpoint (RequireAuthorization)         ← capa Api
    │ pasa
    ▼
AuthorizationBehavior (MediatR pipeline) ← capa Application
    │ procesa [Authorize(Roles = ...)] en el record
    ▼
Handler
```

Cuando se cambia el acceso de un endpoint, **siempre actualizar ambos niveles**. Si solo se actualiza el endpoint, `AuthorizationBehavior` lanzara `ForbiddenAccessException` de todas formas.
